# Threat Model — Maxwell-Daemon

> **Status: DRAFT.** This document is the seed of a fuller threat model started
> as part of issue #797 phase 1 (security headers + CORS scaffolding). It is
> intentionally one page and will be expanded as later phases land
> (HTTPOnly cookies, WebSocket auth, audit log SQLite store, dependency CI,
> secret-redaction whitelist refactor, TLS in Dockerfile).

## Scope

Maxwell-Daemon is the **autonomous AI control plane** that orchestrates agent
tasks. It exposes:

- An HTTP API consumed by `runner-dashboard` and other operators.
- A WebSocket event stream for real-time pipeline observation.
- An ExecutionSandbox that runs validation commands generated by LLM agents.

The most important trust boundary in the system runs through the
ExecutionSandbox.

## Sandbox trust boundary

From the project [`README.md`](../../README.md):

> Validation commands run through an argv allowlist, workspace-root check,
> environment filter, timeout, output redaction, and artifact capture. **The
> current executor uses host subprocesses; it does not provide Docker,
> filesystem, network, process, or resource isolation.** See
> [Security](../operations/security.md) before running untrusted generated
> code.

This means **anything an LLM agent emits as a validation command runs with
the privileges of the daemon process itself**. The mitigations in place are
policy-based, not isolation-based:

| Mitigation               | Layer                                   |
| ------------------------ | --------------------------------------- |
| Argv allowlist           | `maxwell_daemon/executor/`              |
| Workspace-root check     | `maxwell_daemon/executor/`              |
| Environment-variable filter | `maxwell_daemon/executor/`           |
| Wall-clock timeout       | `maxwell_daemon/executor/`              |
| Output redaction         | `maxwell_daemon/audit.py`               |
| Artifact capture         | `maxwell_daemon/core/artifacts.py`      |

A determined attacker who can influence the LLM's generated commands and
defeat the argv allowlist gets the daemon's UID, network namespace, and
filesystem reach. Treat the daemon's host as the security boundary, not the
sandbox.

## Browser-facing trust boundary

The bundled UI (`maxwell_daemon/api/ui/`) and any external dashboard talk to
the daemon via HTTP. Phase-1 of #797 adds defensive response headers:

- `X-Content-Type-Options: nosniff`
- `X-Frame-Options: SAMEORIGIN`
- `Referrer-Policy: strict-origin-when-cross-origin`
- `X-XSS-Protection: 0` (legacy auditor disabled, modern advice)
- `Strict-Transport-Security: max-age=31536000; includeSubDomains` (HTTPS only)
- Optional `Content-Security-Policy` (off by default — opt-in via
  `SecurityHeadersMiddleware(enable_csp=True)`).

A `CORSConfig` Pydantic model is now available under `api.cors` so the
operator can explicitly allow-list cross-origin browser callers. It is
**disabled by default**.

## Out of scope (phase 1)

The following items remain on the roadmap and are **not** addressed by phase 1:

- HTTPOnly / Secure / SameSite cookie hardening for any future cookie auth.
- WebSocket-level authentication (the auth handshake currently relies on the
  HTTP upgrade headers only).
- Persistent audit log storage in a dedicated SQLite store.
- Dependency-vulnerability scanning in CI.
- A whitelist-driven refactor of the secret-redaction logic.
- TLS termination inside the Dockerfile (today TLS is expected at the
  reverse proxy).
